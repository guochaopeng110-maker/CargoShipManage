## Context
系统需要处理后端高频推送的批量分片数据。单次导入规模可达 50,000 条，每 10ms 推送一个分片（100条）。

## Decisions
### 1. 差异化数据流向 (Data Routing)
- **实时批量 (Live Batch)**：视为“快速连续的单条数据”，应并入实时波形展示。通过现有的 1s 批量更新定时器进行处理，以减轻 React 渲染负担。
- **历史导入 (History Import)**：定义为“冷数据引入”，不应污染当前的实时展示窗口。系统仅处理其进度状态，数据持久化由后端完成后，前端通过全局通知引导用户去数据中心查询。

### 2. 状态码语义映射
- 后端推送的 `quality` 为数字类型，前端需要将其标准化：
  - `192` -> `DataQuality.NORMAL`
  - 其他 -> `DataQuality.ABNORMAL`

### 3. 进度追踪算法
- 在 Store 层面维护一个 `Map<batchId, { current, total }>`。
- UI 订阅该 Map 中活跃的批次。
- 当 `current === total` 时，延迟 2 秒销毁该批次状态，并触发成功通知。

## Risks / Trade-offs
- **内存消耗**：如果同时上传多个大文件，WebSocket 缓冲区可能会积压。
- **缓解计划**：前端强制执行单次文件上传限制（50,000条），并限制同时进行的导入任务数量。
