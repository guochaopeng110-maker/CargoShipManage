
   * 目标 (Objective):
      将 DataImportPage
  页面重构为一个功能明确、状态可视、反馈及时的文件上传工具。该页面需遵循新的服务分层架构，并为用户提供一个清晰、可靠的数据导入体验 
  。

   * 核心概念 (Core Concept):
       1. UI/UX - “三步走”布局 (Three-Step Layout):
           * 第一步：文件选择与上传 (File Selection & Upload):
               * 组件: 一个醒目的文件拖放区域或一个标准的“选择文件”按钮，用于接收用户上传的 CSV 或 Excel 文件。
               * 交互: 用户选择文件后，立即显示文件名、大小等基本信息，并提供一个“开始上传”按钮。
           * 第二步：上传与处理状态 (Upload & Processing Status):
               * 组件: 上传开始后，界面切换到一个可视化的进度条，显示文件上传的进度。上传完成后，进度条转变为一个“处理中”的动画（如
                 Spinner），表示后端正在校验和入库数据。
               * 组件: 同时，应有一个 ImportStatusIndicator
                 组件，通过图标和文字（如“等待上传”、“上传中”、“处理中”、“成功”、“失败”）清晰地展示当前任务的状态。
           * 第三步：导入历史与结果 (Import History & Results):
               * 组件: 页面下方始终展示一个带分页的数据表格，用于列出历史导入任务的记录。
               * 内容:
                 表格应包含：任务ID、上传文件名、上传时间、处理状态、成功导入的行数、失败的行数，以及一个“查看详情/错误报告”的链接
                 。
       2. `Store` 与服务层 (Store & Service Layer):
           * 创建 `import-store.ts`:
               * 职责: 管理当前上传任务的状态（uploadProgress, importStatus），以及历史导入任务列表的分页数据 (importHistory,
                 paginationInfo) 和加载状态。
           * 重构 `import-service.ts` (作为 `store` 的 `actions`):
               * `uploadFile(file, onProgress)`: 实现文件上传的核心逻辑。它将调用 api-client 的 POST /api/import/upload 方法。此
                 action 需要能够处理 multipart/form-data，并通过 onProgress 回调来更新 store 中的 uploadProgress 状态。
               * `fetchImportHistory(page)`: 调用 api-client 的 GET /api/import/history 方法来获取历史导入记录。
               * `pollImportStatus(taskId)`: （可选，但推荐）文件上传成功后，后端可能需要异步处理数据。此 action
                 将定期（如每隔几秒）调用 api-client 的 GET /api/import/status/{taskId}
                 接口，直到后端返回“成功”或“失败”状态，然后更新 store 中的 importStatus。
       3. 数据流 (Data Flow):
           * 用户选择文件并点击上传 -> 调用 uploadFile action -> action 通过 api-client 发送文件，并持续更新 store 的进度状态 -> UI
             响应进度变化。
           * 上传完成后 -> （可选）调用 pollImportStatus action 轮询后端处理状态 -> action 持续更新 store 的任务状态 -> UI
             响应状态变化（如从“处理中”变为“成功”）。
           * 页面加载或用户翻页 -> 调用 fetchImportHistory action -> action 通过 api-client 获取历史列表 -> store 更新历史记录状态
             -> UI 表格重新渲染。

   * 预期成果 (Expected Outcome):
       * 一个重构后的 DataImportPage.tsx，它为用户提供了清晰的“三步走”操作流程和实时的状态反馈。
       * 一个功能完备的 import-store.ts 和
         import-service.ts，成功地将文件上传、状态轮询和历史记录查询等复杂逻辑封装起来，并严格遵循了新的分层架构。
       * DataQueryPage 的“筛选-分页列表”模板再次在“导入历史”功能上得到有效复用，进一步证明了其通用性。
       * 用户能够清晰地看到自己的文件上传进度、后端的处理状态，并能方便地追溯每一次导入操作的结果
