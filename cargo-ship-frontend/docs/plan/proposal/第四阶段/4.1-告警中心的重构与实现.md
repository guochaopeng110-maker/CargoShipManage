概念草稿：[阶段四] 任务4.1 - 告警中心 (`AlarmCenterPage`) 的重构与现                                                                                                                                                                                       
   * 目标 (Objective):
      将 AlarmCenterPage
  按照新的分层架构和数据流模式彻底重构，使其成为一个功能完备、体验流畅的告警管理中枢。该页面需要同时处理实时推送的告警和可供查询的 
  历史告警记录。

   * 核心概念 (Core Concept):
       1. 服务与状态层重构 (Service & Store Refactoring):
           * `alarms-store.ts` 职能划分:
               * 实时状态: store 内部需维护一个 activeAlarms 状态，由 realtime-service 推送的 alarm:push, alarm:batch 和
                 alarm:update (例如，告警被确认后) 等事件实时驱动更新。
               * 历史状态: store 需增加用于管理历史告警查询的独立状态，完全复用 DataQueryPage 模板的模式，包含 historicalAlarms: {
                 items: [], total: 0, page: 1 } 和 queryStatus: 'loading' | 'success' | 'error' 等。
           * `alarms-service.ts` (作为 `store` 的 `actions` 实现):
               * 实现 fetchHistoricalAlarms(filters, page) 动作，该动作负责调用 api-client 中的 GET /api/alarms/history 方法。
               * 实现 acknowledgeAlarm(alarmId) 动作，该动作负责调用 api-client 中的 POST /api/alarms/{alarmId}/ack
                 方法，用于告警确认。
       2. UI/UX - 双模视图设计 (Dual-Mode View):
           * 页面主体采用 Tabs 组件 进行视图切换，分为“实时告警”和“历史告警”两个标签页。
           * “实时告警” 视图:
               * 展示一个数据表格，数据源为 alarms-store 中的 activeAlarms 状态。
               * 此列表应能自动、实时地响应WebSocket推送，无需用户手动刷新。
               * 表格的每一行都应包含一个“确认 (Acknowledge)”按钮。
           * “历史告警” 视图:
               * 完全复用 DataQueryPage 确立的“筛选-分页列表”模板。
               * 包含一个筛选区（可按设备、告警等级、日期范围等查询）和一个带分页功能的数据表格。
               * 数据由用户点击“查询”按钮后，通过调用 fetchHistoricalAlarms 动作来按需加载。
       3. 闭环数据流 (Closed-Loop Data Flow):
           * 告警确认流程: 用户在“实时告警”视图中点击“确认”按钮 -> 调用 acknowledgeAlarm(alarmId) 动作 -> api-client
             发送POST请求到后端 -> 后端处理后，通过WebSocket推送一个 alarm:update 事件 -> realtime-service 接收并分发给
             alarms-store -> alarms-store 从 activeAlarms
             列表中移除该告警，UI自动更新。这个闭环确保了操作的即时反馈和状态的一致性。

   * 预期成果 (Expected Outcome):
       * 一个重构完成的 alarms-store.ts，它能够优雅地同时管理实时数据流和用户驱动的API请求数据。
       * 一个全新的 AlarmCenterPage.tsx，通过Tabs组件清晰地分离了实时监控和历史追溯两种用户场景。
       * 一个功能完整的告警管理工作流被成功实现，展示了新架构下，用户操作、API请求和WebSocket实时更新如何协同工作的最佳实践。
       * 第三阶段确立的“筛选-分页列表”模式得到了成功的复用，验证了其模板价值。
