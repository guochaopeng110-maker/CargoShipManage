 概念草稿：[阶段四] 任务4.2 - 系统管理模块（设备/用户/角色）的模板化重构

   * 目标 (Objective):
      将设备管理 (DeviceManagementPage)、用户管理 (UserManagementPage)和角色与权限
  (RoleManagementPage),阈值管理(ThresholdManagementPage)这四个页面，统一使用一个标准的、基于api-client的“CRUD页面模板”进行重构。此举旨在最大化代码复用，并为应用中所
  有管理类页面建立一个统一、高效的开发范式。

   * 核心概念 (Core Concept):
       1. 统一的“CRUD页面模板” (Standard CRUD Page Template):
           * 布局:
               * 顶部 - 操作栏 (Action Bar): 包含一个“新增”按钮和一个全局搜索框。
               * 主体 - 数据表格 (Data Table):
                 使用一个功能强大的、带分页和排序功能的数据表格组件来展示列表。表格的最后一列固定为“操作”列，包含“编辑”、“删除”等图
                 标按钮。
           * 交互:
               * 点击“新增”或“编辑”按钮，会弹出一个模态框 (Modal/Dialog)，内部包含用于数据录入的表单。
               * 点击“删除”按钮，会弹出一个确认对话框 (Confirm Dialog)，防止用户误操作。
       2. 专用的、模块化的`Store` (Dedicated & Modular Stores):
           * 为每个管理模块创建独立的 store，例如：
               * equipment-store.ts
               * auth-store.ts
               * threshold-store.ts
           * `Store` 的统一职责:
               * 状态管理: 负责管理列表数据、分页信息 (total, page, pageSize)、加载状态 (loading) 以及当前模态框的状态（如
                 isModalOpen, editingItem）。
               * `Actions` 封装: 封装所有CRUD操作。这些 actions 将是调用 api-client 的唯一入口。例如，user-management-store 会有
                 fetchUsers(page, query), createUser(data), updateUser(id, data), deleteUser(id) 等 actions。
       3. 服务层与API调用 (Service Layer & API Calls):
           * 数据流: 页面组件自身是“哑”的，它仅从对应的 store 中消费状态并调用 actions。
           * `Actions` 实现: store 中的每个 action 都会直接调用由 openapi-typescript-codegen 生成的、类型安全的 api-client
             方法。例如, fetchUsers 会调用 api.getUsers({ page, query })。
           * 乐观/悲观更新:
               * 对于删除操作，可采用“乐观更新”：UI上先移除该行，再发送API请求。如果请求失败，则将该行恢复并提示用户。
               * 对于新增/编辑操作，采用“悲观更新”：提交表单后，显示加载状态，直到API请求成功返回后，才关闭模态框并刷新列表。

   * 预期成果 (Expected Outcome):
       * DeviceManagementPage, UserManagementPage, RoleManagementPage,ThresholdManagementPage
         四个页面被成功重构，它们共享几乎完全相同的布局和交互逻辑，仅在表单字段和表格列定义上有所不同。
       * 一套清晰的、可复用的“CRUD页面开发模式”被确立下来，包括专门的store结构、actions定义以及与api-client的交互方式。
       * 项目代码的复用率和开发效率得到显著提升，未来任何新增的管理页面都可以通过复制此模板快速实现。
       * 通过对auth-service（涉及用户/角色）等相关高层服务的重构，彻底贯彻了新的服务分层封装原则。
